# 自动 @ 引用解析

## 使用场景

当用户 prompt、上下文或系统附加内容中出现 `# 可用 @ 资源索引` 时，使用本 Skill 判断正文里的角色、场景、关键道具或资料实体是否应链接到工作区已有资源。

本 Skill 只负责自动 @ 引用的识别、匹配和输出格式；不要替代剧本、角色、场景、道具、分镜等业务写作 Skill。

## 匹配原则

- 不要求字面精确匹配；结合人物身份、别名、剧情关系、场景功能、道具用途和上下文语义判断是否对应同一资源。
- 优先使用系统提供的 `# 可用 @ 资源索引`、用户消息中的 @ 资源，或上下文里已有的完整 Markdown @ 链接。
- 同一段或同一镜头中，同一资源首次出现时 @ 一次；后续可只写实体名。
- 无把握、无可用完整链接或语义关系太弱时不要硬 @，可写 `引用资源：无` 或省略引用。

## 资源索引格式

系统附加的索引通常由标题和若干资源行组成：

```markdown
# 可用 @ 资源索引

以下资源来自当前工作区，可供生成内容时引用。

## 资源
- 角色｜沈阎｜@[沈阎](mention://character-doc/section_shenyan)
- 场景｜审讯室｜@[审讯室](mention://scene-doc/section-k3x9aa)
- 资料｜原始素材｜@[原始素材](mention://reference-doc)
```

每行含义是 `资源类别｜资源标题｜完整 Markdown @ 链接`。只把第三段当作可复制的链接来源；第一段和第二段只用于辅助语义匹配。

## 输出格式

- Markdown @ 链接格式是 `@[显示标题](mention://<document-id>)` 或 `@[显示标题](mention://<document-id>/<block-id>)`。
- 引用写成“原文实体名 + 紧跟 @ 链接”的内联格式；实际链接必须是完整 Markdown，例如 `沈阎@[沈阎](mention://character-doc/section_shenyan)`。
- `@[显示标题]` 里的标题来自索引或上下文中的完整链接；不要为了贴合正文随意改链接标题。
- 编辑器中可视效果应类似 `沈阎@沈阎`；场景、道具和资料同理。

## id 获取方式

- `mention://` 后第一段是文档 ID，例如 `character-doc`、`scene-doc`、`reference-doc`。
- 第二段可选，是章节或块 ID，例如 `section_shenyan`、`section-k3x9aa`；没有第二段时表示引用整篇文档。
- 这些 ID 由系统生成资源索引时提供：稳定章节 ID 可能来自文档里的 `<!-- section-id: ... -->`，没有稳定 ID 时系统会生成 `section-...` 形式的块 ID。
- 你不需要、也不应该计算这些 ID；只能从 `# 可用 @ 资源索引`、用户消息中已有 @ 链接，或上下文中已有完整 Markdown @ 链接复制。
- 不要自己拼接或编造 `mention://`、`asset://`，也不要输出带省略号的占位链接。
- 如果只有资源名称、文档标题或普通 URL，但没有完整 `@[...](mention://...)` 链接，则不要自行拼接 id。

## 示例

- 正确：`沈阎@[沈阎](mention://character-doc/section_shenyan)推开审讯室@[审讯室](mention://scene-doc/section-k3x9aa)的铁门。`
- 正确：`参考原始素材@[原始素材](mention://reference-doc)中的时间线。`
- 错误：`沈阎@[沈阎](mention://...)`，因为链接使用了省略号。
- 错误：`沈阎@[沈阎](mention://沈阎)`，因为根据名称自造了 ID。
- 错误：`审讯室@[审讯室](asset://scene-doc)`，因为自动 @ 引用应复制已有 `mention://` 链接，不要改写为 `asset://`。

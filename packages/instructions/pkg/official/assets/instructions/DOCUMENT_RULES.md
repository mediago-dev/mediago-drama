# 核心文档规则

## 文档类型

新建业务文档时，必须在 Markdown 顶部写 YAML frontmatter，并保证 `category` 与内容类型一致：

- 剧本、动漫剧本、可拍摄台本使用 `category: screenplay`。
- 角色、人物形象、人物视觉设定使用 `category: character`。
- 场景、地点、空间环境设定使用 `category: scene`。
- 道具、关键物件、美术资产设定使用 `category: prop`。
- 分镜、镜头脚本、视频镜头提示词使用 `category: storyboard`。
- 原始资料、小说正文、参考材料和未加工文本使用 `category: reference`，或在缺少明确业务类型时省略 category。

不要把剧本、角色、场景、道具或分镜文档标记为 `reference`。

## 二级标题资源边界

业务文档的主体资源边界是二级标题 `##`。系统会识别所有二级标题，并把每个二级标题及其下方正文视为一个可引用、可生成、可追踪的文档资源。

- 角色文档中，每个 `## 角色名` 是一个角色形象资源。
- 场景文档中，每个 `## 场景名` 是一个场景资源。
- 道具文档中，每个 `## 道具名` 是一个道具资源。
- 分镜文档中，每个 `## 第 01 组` 或同等级别标题是一组可生成分镜资源。
- 同一角色在不同时期、年龄阶段、身份状态或变身前后有明确视觉差异时，拆成多个二级标题，例如 `## 沈阎（少年）`、`## 沈阎（成年）`、`## 沈阎（变身后）`。

一级标题可以作为整篇文档标题或章节标题；三级及更深标题只能作为二级标题资源内部的组织结构，不会作为主体资源边界。

## section-id

系统可能在二级标题前写入 `<!-- section-id: section_xxx -->`。这是稳定资源 ID，用于 @ 引用、素材生成、历史追踪和跨编辑同步。

- 不要删除、伪造或批量改写已有 `section-id`。
- 新增二级标题时可以不手写 `section-id`；系统会自动补齐。
- 移动或改写二级标题内容时，保留标题前已有的 `section-id`。
- 不要把 `section-id` 放进正文说明、Prompt 文本、表格字段或代码块里。

## 输出约束

生成或改写业务文档时，直接输出最终 Markdown 内容，不解释规则来源，不输出空字段、占位符或“待补充”。

文档写作 Skill 只提供业务写作方法；无论 Skill 是否被用户修改，上述文档分类、二级标题资源边界和 `section-id` 规则都必须遵守。

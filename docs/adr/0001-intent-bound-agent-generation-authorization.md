# ADR-0001：Agent 图片与视频采用意图绑定的一次性授权

- 状态：Proposed
- 日期：2026-07-16
- 决策范围：Agent 通过 Generation MCP 创建图片、视频任务

## 背景

当前 Agent 在调用 `generate_media` 或 `generate_media_batch` 前，会要求用户提交
`generation_plan`。服务端已经核验 selection 所属的 project/session/run，以及模型路由、
参数、参考图、提示词包和提示词优化设置。

现有边界仍存在两类缺口：

1. run ID 为空时会被当作“兼容模式”，图片和视频不再要求确认；生产 HTTP/stdio
   接入都可能因为上下文丢失进入该分支。
2. 确认只绑定生成设置，不绑定实际要生成的提示词、目标资源、批次数量与顺序；同一
   selection 也可重复使用。
这意味着“用户确认过参数”还不是对一次具体副作用的完整授权。

## 决策

Agent 图片和视频生成改为 **意图绑定、一次使用、可幂等重放** 的授权模型。

### 1. 显式区分调用模式

Generation MCP 的调用方模式必须由构造器显式声明：

- `agent`：图片和视频创建必须具备 project、session、run 与 selection store；任何
  上下文缺失都 fail closed。
- `manual`：仅供明确的非 Agent 集成使用，不再通过“run ID 是否为空”推断。

项目工作台已有的 HTTP 生成与任务操作继续由用户界面承担授权边界，不接入 Agent selection。
音频、文本维持现有策略，不在本次决策范围内。

### 2. 确认同时绑定设置与生成意图

`generation_plan` 除可编辑的 `generation_settings` 外，还保存不可变的
`GenerationPlanIntent`：

- 操作类型：单项创建或批量创建；
- 有序子项：item ID、kind、基础 prompt、资产标题；
- 目标：project、生成会话、scope、document/section/documentContext/resourceType；
- 每项固定参考资产和通知目标；
- 批次标题、数量与顺序。

用户卡片必须显示“将生成什么、生成几项、写到哪里”的摘要和可展开明细。点击确认后，
提交的设置快照与不可变意图共同构成授权内容。

实际 MCP 请求规范化后必须与授权完全一致。允许的动态合并只有：

- 路由、params、公共参考图、提示词包和提示词优化来自用户提交的
  `generation_settings`；
- 每项 prompt、目标与固定参考图来自 `GenerationPlanIntent`；
- 服务端生成的 task/batch ID 和纯后端默认值不参与比较。

Agent 请求继续禁止 `referenceUrls`、`referenceBindings` 和模型覆盖字段，避免绕过已确认
的资产与路由快照。

### 3. 一次确认只授权一次副作用

- 单项确认授权一次 `generate_media`。
- 批量确认授权一次完整 `generate_media_batch`，而不是让每个子项各自复用 selection。
- 批次的 item 数量、ID、顺序与每项内容全部参与指纹。
- selection 被原子 claim 后不可再次发起供应商请求。

服务端为规范化后的完整请求计算稳定指纹，并以 compare-and-swap 原子 claim：

- 首次相同请求：获得 claim，允许提交；
- 同指纹且已有结果：直接返回首次结果，不重复生成；
- 同指纹但结果尚未落库：返回“处理中或结果未知”，不得重提；
- 不同指纹：拒绝，要求重新确认。

一旦进入可能产生外部副作用的调用，失败或进程中断也不释放 claim。这个选择优先避免
重复扣费；用户可在工作台处理既有任务，或通过新的 generation plan 发起另一份创建请求。

### 4. 已确认动作的后台履约与工作台操作不扩展 Agent 授权

Generation MCP 只公开单项创建与批量创建，不向 Agent 暴露任务查询、重试、主动轮询或选片工具。
任务查询、HTTP 重试能力、后台轮询和选片继续由工作台与后台服务承担，不复用 Agent selection。

已确认创建动作的以下后台履约行为不再次弹窗：

以下行为不产生新的生成授权，因此不再次弹窗：

- 已创建任务的供应商后台提交、轮询、下载、缓存和完成通知；
- 查询任务状态和查看结果；
- 对已有结果执行不触发新生成的选片或展示操作；
- 对同一请求的幂等结果重放。

Agent 若要新建供应商任务或增加输出数量，必须使用一个新的、未消费的 generation plan；
工作台 HTTP 操作继续使用其自身的用户操作边界。

## 结果与取舍

### 正面结果

- 上下文缺失不再降级成免确认。
- 用户确认的对象从“参数”提升为“一次具体生成动作”。
- 双击、相同创建调用的技术重放和并发调用不会重复创建任务。
- 单次批量确认仍保留批量体验，不会为每个镜头弹一次窗。
- 审计记录可回答谁在何时确认了哪一批内容，以及对应的 task/batch 结果。

### 代价

- selection 需要新增意图、claim 指纹和结果快照字段。
- Agent 表单事件与前端需要展示不可变意图摘要。
- 生成适配层需要在副作用前完成规范化、claim，并在成功后保存可重放结果。
- 发布前已经提交但不含 intent 的历史 generation plan 不能继续授权生成，需要重新确认。

## 未采用方案

### 仅修复空 run ID

能关闭最明显的绕过，但 prompt、批次数量和重复使用仍未绑定，不能满足“一次人工确认对应
一次具体生成”。

### 只在提示词中要求 Agent 不要跳过确认

提示词是行为指导，不是服务端授权边界；无法覆盖错误客户端、上下文丢失、并发调用或
相同创建调用的技术重放。

### 每个批次子项单独确认

安全性足够，但会把批量生成退化成 N 次交互。完整批次作为一个有序原子意图即可同时
满足可理解性与可用性。

### claim 失败后自动释放

无法可靠判断供应商是否已收到请求，自动释放可能造成重复任务和重复扣费。因此采用
fail-safe：结果未知时不自动重提。

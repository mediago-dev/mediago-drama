# mgmd 格式规范(草案 v2)

> 状态:核心方向已确认(2026-07-09)。本版取代 v1 的"属性语法 `{#id}`"方向——改为**文件里用 HTML 注释承载身份(对普通渲染器隐形)、编辑器内存里用节点属性承载身份(结构操作不丢)**。

mgmd(MediaGo Markdown)是 MediaGo Drama 的文档持久化格式。它是 **CommonMark + GFM 的薄超集**:任何 mgmd 文档都是合法的标准 Markdown,可被任意 Markdown 工具打开、且 mgmd 的扩展语义对普通渲染器**隐形**(不显示成杂字);mgmd 感知的一方(编辑器、服务端、MCP、LLM)则能额外识别其中的结构化语义。

## 1. 目标与非目标

**目标**

- 为章节/块提供**稳定身份(section-id)**,支撑 @ 引用、评论锚点、素材生成、历史追踪、块级链接。
- 稳定身份对"改名、移动、复制、剪切"都成立。
- 扩展语义对普通 Markdown 渲染器**隐形**。
- 三端(前端 TS / Go 服务端 / MCP)解析与写入行为**严格一致**,以共享测试语料强制。

**非目标**

- 不做块树 canonical(Notion / 飞书模式)。**Markdown 文本是落盘时的唯一事实来源。**(会话内协同时的 canonical 见 §12。)
- 不承载纯视觉样式的持久化(颜色、对齐)。样式维持编辑器会话内实现,不进 md(见 §9)。
- 不保证扩展语义在外部渲染器里"生效",只保证**隐形 + 优雅降级**。

## 2. 已确认的核心设计决策

| # | 决策 |
|---|------|
| 身份语法 | 文件里用 `<!-- section-id: section_xxx -->`(HTML 注释,隐形)。**不用 `{#id}`**。 |
| 内存表示 | 编辑器内 section-id 是 **heading 节点的属性**,不是独立节点;注释只在读写文件的边界折叠/还原。 |
| 稳定性 | **全量补写**——保存时给每个二级标题都补 section-id,消灭"无 ID 靠标题哈希"的不稳定态。 |
| 补写位置 | **前端 + 服务端都补**(前端即时、服务端兜底);两端逻辑必须一致,由 §8 一致性语料强制。 |
| 复制/剪切 | 移动保留 ID;复制的副本粘贴时去重换新 ID(§7)。 |
| 剪贴板 | 粘到外部软件不带 ID(干净);粘回 mediago-drama 保留 ID(§7)。 |
| 通用元数据 | 定义"HTML 注释承载隐形元数据"的通用机制(§L3),v1 只注册 section-id,不塞其他键。 |
| 音视频块 | 保持现有标签链接形式(§L4),v1 不迁移。 |
| 样式(对齐) | 不做 md 持久化;维持现有会话内实现(§9)。 |
| 导出 | 默认保留注释(隐形无损),另给"剥净"选项(§11)。 |

## 3. 设计原则

1. **隐形优先**:扩展语义在普通渲染器里不得显形;唯一允许的隐形承载是 **HTML 注释** 与 **YAML front matter**。
2. **身份存注释、内存用属性**:文件里身份是注释(隐形);内存里身份是节点属性(结构操作跟随节点、不掉队)。两者只在序列化边界转换。
3. **降级契约**:每个注册扩展必须声明四栏降级行为(§10),缺一不得注册。
4. **未知保留**(向前兼容核心):解析器遇到**未注册**的 mgmd 注释键 / scheme 时,必须原样保留并原样输出,不得丢弃或报错。旧客户端打开新文档不丢数据。
5. **章节 ID 全量化**:二级标题在保存时一律自动补 section-id,使"改名/移动不变身份"成为无条件保证,并淘汰三端各自的派生 ID 算法。

## 4. 语法分层

### L0 基线:CommonMark + GFM

启用:表格、删除线、任务列表、自动链接。标题用 ATX 风格,层级锁 1–4 级,二级标题为章节资源边界(与现有 DOCUMENT_RULES 一致)。

### L1 文档头:YAML front matter(隐形)

**现状**:front matter **已在用**——文档层元数据 `id / title / category / version` 已经写在文件头,由文档存储层管理(不进编辑器正文):

```markdown
---
id: doc-file-94cfb4731c570bc5
title: 角色-猴子
category: character
version: 28
mgmd: 1
---
```

- `mgmd` 为新增的保留键,声明规范版本。无 `mgmd` 键的文档按当前版本 + 遗留兼容(§13)解析。
- 其余文档级键遵循"未知保留"。

### L2 身份:section-id(HTML 注释,隐形)

文件里,section-id 是标题**上方**的 HTML 注释(与标题之间可隔一空行;关联规则是"从标题往上找第一个非空行"):

```markdown
<!-- section-id: section_dd18ed89940660bb -->

## 林墨（高三 · 契约前）
```

- ID 字面量:`section_[A-Za-z0-9_-]+`(现状为 `section_` + 随机 16 位),文档内唯一。
- **全量补写(现状:服务端已实现)**:`POST /workspace/sections/reconcile` → `ensureDocumentSectionAnchors` 已给每个缺 id 的二级标题补注释并存回。阶段 1 补齐的是**前端一致 + 确定时机 + 三端语料**(§8),不是从零造。
- **注释不是内存里的独立节点(目标态)**——见 §6:解析时折叠进后续标题的属性,序列化时从标题属性还原成注释。**现状**是独立的 `sectionIdAnchor` 节点,阶段 1 改为 heading 属性。

### L3 通用隐形元数据机制(定义,v1 不注册键)

所有隐形块级元数据统一走 HTML 注释,挂在目标块前一行:

```markdown
<!-- mgmd: key=value key2="带空格 value" -->
```

- v1 **不注册任何 `mgmd:` 键**;机制先定义好,将来加块级元数据(如场景属性、生成状态)无需再发明语法。
- `section-id` 是本机制的一个已注册特例(保留其现有 `<!-- section-id: xxx -->` 写法)。
- 未注册键遵循"未知保留"。

### L4 标签链接 + 自定义 URI scheme

行内引用与媒体块沿用现有语法——标准链接 + 私有 scheme / 标签前缀,本身即合法 Markdown:

```markdown
@[角色-猴子](mention://doc_xxx)
@[1-1 教室](mention://doc_xxx/section_a1b2c3)
@[分镜图](asset://asset_xxx?kind=image&mimeType=image/png)
[章节音频：1-1 配音](</api/resource/ep1-s1.mp3>)
```

- **音视频块**:保持标签链接(`章节音频：` / `章节视频：` 前缀),v1 不迁移。理由:它外部降级为可点击链接,比注释更友好。未来若需挂封面/时长等属性,再迁到 §L3 注释机制。
- **注册 scheme(v1)**:

| scheme | 含义 |
|--------|------|
| `mention://<docId>[/<sectionId>]` | 文档 / 章节引用 |
| `asset://<assetId>?kind=&mimeType=&url=` | 素材引用 |

## 5. 目标 ↔ 内存 ↔ 文件 的三态

```
文件(落盘 canonical)            编辑器内存(会话内)
<!-- section-id: s_a1b2 -->  ─解析→  heading { attrs.sectionId: "s_a1b2", ... }
## 1-1 教室                          ↑ 注释被折叠进标题属性,内存里没有独立注释行
                             ←序列化─
```

- **打开文件**:解析器读到 section-id 注释,把它折叠成紧随其后标题的 `sectionId` 属性,**不保留独立注释节点**。
- **保存文件**:序列化器从每个标题的 `sectionId` 属性,还原出 `<!-- section-id -->` 注释放到标题上一行。
- 因此**编辑期间不存在"上一行的注释"**——身份就在标题节点里,拖动/剪切随节点走。

## 6. 编辑器内存模型

- section-id = heading 节点的 `sectionId` 属性(废弃独立的 `sectionIdAnchor` 节点)。
- heading 的 `renderHTML` 输出隐形的 `data-section-id`,`parseHTML` 读回——这样剪贴板 `text/html` 与编辑器往返都带身份(§7)。
- 补写发生在保存路径(前端)与入库路径(服务端);两端行为由 §8 语料对齐。

## 7. 剪贴板行为(多 flavor)

复制/剪切一个块时,系统剪贴板同时写入多种格式,各有不同策略:

| flavor | 内容 | 结果 |
|--------|------|------|
| `text/html` | 节点 HTML,含隐形 `data-section-id` | 粘回 mediago-drama → 还原身份 |
| `text/plain` | 纯文本 / 无 mgmd 元数据的 markdown,**不含 section-id 注释** | 粘到外部软件 → 干净,无杂 |

规则:

- **粘到别的软件**:对方取 `text/plain` → 拿到干净内容,没有 section-id;若对方取 `text/html`,`data-section-id` 是隐形属性,不显示。
- **粘回 mediago-drama**:走 `text/html` / 内部格式 → 身份保留。
- **移动 vs 复制的去重**:粘贴时若 section-id 与当前文档已有的撞车 → 给新来的重生成(首个保留)。剪切(移动)时 ID 已不在文档中 → 不撞车 → **保留原 ID**;复制(副本)时原件仍在 → 撞车 → **副本换新 ID**。

## 8. 解析 / 写入的三端一致性

| 端 | 实现 | 说明 |
|----|------|------|
| 前端 | `@tiptap/markdown`(marked)自定义 tokenizer | 读注释折叠进 heading 属性;序列化时还原注释;保存时全量补写 |
| Go 服务端 | goldmark(读注释)+ 补写逻辑 | 入库兜底全量补写 |
| MCP | 同 Go(goldmark) | 与服务端共用解析包,消除第三份拷贝 |

**一致性约束(硬)**:`docs/mgmd-conformance/` 维护共享测试语料——输入 md → 期望结构 → 期望序列化输出,覆盖:补写 ID 格式、注释放置位置、去重规则、往返稳定性。**前端与服务端的补写逻辑必须跑同一套语料,行为以语料为准**,不允许各写各的正则(对应 ◆1 "务必逻辑一致")。

## 9. 样式(对齐 / 缩进)

- 对齐/段落缩进等纯视觉属性**不做 md 持久化**——已验证它们无法在标准 markdown 中表示。
- 维持**现有会话内实现**(编辑器内可见、刷新还原);列表缩进因是嵌套列表可正常持久化。
- 若将来判断对齐对剧本有足够价值,再走 §L3 注释机制承载(隐形 + 可持久化 + 需自渲染)。

## 10. 降级矩阵(注册扩展必填)

| 扩展 | mgmd 应用内 | 普通 md 渲染器 | 纯文本 | 导出剥净时 |
|------|------------|---------------|--------|-----------|
| `<!-- section-id -->` | 章节身份(内存 attr) | **隐形** | 可见的注释文本 | 剥除 |
| `<!-- mgmd: k=v -->`(预留) | 隐形块元数据 | **隐形** | 可见的注释文本 | 剥除 |
| `[章节音频：…](<src>)` | 播放器卡片 | 可点击链接 | 字面文本 | 保持原样 |
| `@[...](mention://…)` | 引用胶囊 | `@` + 死链接(可见标题) | `@[标题](uri)` | 转为纯文本标题 |

## 11. 导出策略

- **默认保留** section-id / mgmd 注释——注释本就隐形,保留可让文件回到 mediago-drama 时无损往返。
- 另提供**"剥净"导出**:剥掉所有 mgmd 注释、把 mention 转纯文本,产出最干净的标准 md 供外发。

## 12. 面向协同的架构(预留方向,暂不实现)

未来支持桌面 + 移动端实时协同时,采用**混合架构**,不翻成 Notion/飞书式树 canonical:

- **落盘 / git / 给 LLM**:markdown 仍是 canonical(保住 LLM-native 与可移植性)。
- **一次协同会话进行中**:Yjs 绑定的 ProseMirror 树是 canonical(实时合并、光标、评论);会话开始 md→树,存盘时树→md。
- **桥梁**:稳定 section-id / 块 ID(即本规范的工作)——让评论锚点、远端光标、以及"树↔md 往返不乱"都成立。
- **降级保证**:markdown-canonical 是地板,任何协同子能力做不成,都只退回"今天的单人架构 + 非实时协同",**不伤 AI 编辑核心**。
- 本规范为此**预留**:身份机制未来可从"仅二级标题"扩展到"每个块"。

## 13. 遗留兼容 与 发布时序

**遗留兼容(只读识别,长期保留):**

| 遗留 | 处理 |
|------|------|
| `section-<hash>` 旧格式 ID / 各端派生 ID | 全量补写后退役,仅保留只读识别(引用仍可命中) |

**发布顺序(桌面应用硬约束):**

1. **阶段 0**:建 `docs/mgmd-conformance/` 语料,用现状语法锁定快照。
2. **阶段 1**:三端**双读**(新旧都认)+ 内存改 heading 属性(替代独立 anchor 节点,解决复制/剪切丢 id)+ 前端也做全量补写(服务端已有,补前端一致 + 确定时机)。发布后线上行为不变,可观察一个版本。
3. **阶段 2**:切写入(序列化统一走注释 + 全量补写),DOCUMENT_RULES 同批更新。
4. **阶段 3**:收尾——删派生 ID 代码、删独立 anchor 节点。

每步独立可回滚。**双读版本必须先覆盖存量客户端,隔至少一个版本周期再切写入。**

## 14. 版本与演进

- **新增** mgmd 注释键 / scheme:不升版本,旧解析器按"未知保留"安全通过。
- **变更/移除**已注册语义、或引入旧解析器无法保留的语法:升 `mgmd` 主版本,front matter 声明。
- 注册即承诺:进注册表的扩展需同时提交降级矩阵行 + 三端一致性语料。

## 15. 示例文档

```markdown
---
mgmd: 1
---

# 第一章 御灵之门

<!-- section-id: section_a1b2c3 -->
## 1-1 日 内 御灵一中 高三五班教室

人物:@[林墨](mention://doc_chars/section_linmo)、叶初夏(OS)

下课铃响过,教室里仍是嗡嗡一片。

[章节音频：1-1 配音](</api/resource/ep1-s1.mp3>)

<!-- section-id: section_d4e5f6 -->
## 1-2 日 内 御灵一中 高三五班教室

黑靴踏进门,高挑的黑色制服身影停在讲台前。
```

> 打开到编辑器后,两条 `<!-- section-id -->` 注释各自折叠进下方标题的 `sectionId` 属性,内存里不再有独立注释行;保存时再还原。

## 附录 A:设计考量对照(设计一个 md 格式需要考虑什么)

> 从"设计一个 markdown 文档格式本该考虑哪些维度"出发,逐条对照:现在怎么设计的 / 打算怎么设计 / 为什么。**`❌ 没有考虑到`** 标出当前的真空白,待决。

### A.1 基线与外部兼容

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 基线语法 | CommonMark + GFM | 不变 | 可移植、LLM 友好、生态成熟 |
| 对普通渲染器的可见性 | 注释隐形、媒体是链接 | 隐形优先,扩展只用 HTML 注释 + front matter 承载 | 文件外发到 GitHub/txt 不露杂字(硬需求) |
| 与标准 md 的往返 | 每次编辑已经过 parse/serialize | 加固 + 归一化 + 语料锁定 | 协同、diff、LLM 改写都依赖往返稳定 |

### A.2 身份(identity)—— 核心

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 谁有 ID | 仅二级标题 | v1 仍仅 H2;为协同预留"每个块" | 剧本按章节/场景切,H2 是资源边界 |
| ID 格式 | `section_<随机16位>` stored | 不变 | 随机值与标题解耦,改名不影响 |
| 生成时机 | 服务端 reconcile 时全量补(懒触发) | 前端也补 + 时机确定 + 两端逻辑一致 | 消除"新标题未补的窗口期"、三端一致 |
| 改名稳定性 | 已补的标题改名不变 | 不变 | id 在注释里,改标题正文不碰它 |
| 复制/剪切稳定性 | 丢 id(id 在独立 anchor 节点,带不走) | 改 heading 属性 → 随节点走不丢 | id 长在标题节点上才健壮 |
| 冲突去重 | 无(复制根本不带 id) | 粘贴撞车重生成(移动保留、副本换新) | 移动 vs 复制语义不同 |
| 承载位置(文件/内存分离) | 文件=注释,内存=独立 anchor 节点 | 文件=注释,内存=heading 属性 | 隐形(文件)+ 健壮(内存)两头都要 |

### A.3 扩展机制与元数据

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 挂载入口数 | 散(front matter / 注释 / 标签链接 / scheme) | 锁 4 个入口 | 入口越少越可控 |
| 通用块元数据机制 | 无(section-id 是唯一特例) | 定义 `<!-- mgmd: k=v -->` 机制,v1 不塞键 | 未来加元数据不再逐个发明语法 |
| 自定义块(音视频) | 标签链接 `[章节音频：x](<src>)` | 保持,暂不迁 directive | 外部降级成可点链接比注释友好 |
| 行内引用(mention/asset) | `@[..](mention://..)` scheme | 保持 | 标准链接 + 私有 scheme 本就正规 |

### A.4 表示与序列化

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 内存 vs 落盘 | 落盘 md;内存 ProseMirror 树 + goldmark AST(均派生) | 不变 | md-canonical 是护城河 |
| 往返无损 / 归一化 | 未系统保证 | 归一化 + 一致性语料锁定 | 协同合并、diff 都要求无 churn |
| 导入外部 md 的归一化 | ❌ 没有考虑到(导入时是否补 id / 加 front matter) | 待定 | 用户从外部拖入 .md 时的行为未定义 |

### A.5 多参与者

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 人(编辑器) | 注释隐形节点(display:none) | heading 属性,更隐形不可误删 | 用户不该看到/误删身份 |
| LLM | DOCUMENT_RULES 要求保留注释 | 指令 + 补写兜底 + 程序按 id 重贴 | LLM 改的是文本级,易把注释落下 |
| 多端解析一致 | 前端派生 vs 服务端补写,不一致 | 一致性语料强制 | 三端行为必须一致 |
| 外部工具互操作 | 复制出去丢 markdown、不带 id | 剪贴板多 flavor(见 A.6) | 见下 |

### A.6 剪贴板与互操作

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 复制到外部软件 | text/plain = 纯可见文字(连 markdown 都丢) | text/plain = 干净 markdown(剥注释) | 粘到别的 md 编辑器要能用、且无 id |
| 复制回本应用 | 不保留 id | text/html 带隐形 `data-section-id` | 内部往返身份不丢 |
| 导出干净 md | ❌ 没有考虑到(无导出器) | 导出器:默认保留 / 一键剥净 | 外发场景 |

### A.7 治理与演进

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 版本声明 | ❌ 没有 | front matter `mgmd:1` + 升版规则 | 破坏性变更要能标记 |
| 向后兼容 | 服务端有 legacy 派生 id 只读识别 | 双读(新旧都认) | 旧文档不炸 |
| 向前兼容(未知保留) | ❌ 没有正式约定 | 未知注释键原样保留往返 | 旧客户端打开新文档不丢数据 |
| 降级契约 | ❌ 没有矩阵 | 每个扩展必填四栏矩阵(§10) | 进注册表的门槛 |
| 一致性保证 | ❌ 没有语料 | conformance corpus,三端共跑 | 规范以语料为准,不以文字为准 |

### A.8 样式、富媒体、架构

| 考虑点 | 现在怎么设计 | 打算怎么设计 | 为什么 |
|--------|-------------|-------------|--------|
| 视觉样式(对齐/颜色) | 对齐仅会话内(不持久化),颜色无 | 不做 md 持久化,维持现状;要做走注释机制 | 标准 md 表达不了,且剧本价值低 |
| 图片 | `![](apiResourceURL)` | 不变 | 标准 md 图片够用 |
| 表格/列表往返 | 依赖 `@tiptap/markdown` 默认 | ✅ 阶段 0 语料已验证:往返**幂等、不丢失**(表格首次会归一化列宽,之后稳定) | 语料 04/05 覆盖 |
| Canonical 归属 | md 落盘 canonical | 不变;协同会话内 Yjs 树 canonical | md 是 LLM 护城河 |
| 协同编辑 | ❌ 没有(单人) | 预留:Yjs 混合,稳定 id 做桥 | 桌面 + 移动实时协同需求 |
| 版本历史 / diff 粒度 | DocumentHistoryPanel(文本版本) | ❌ mgmd 层未设计 diff 粒度 / 与块 id 的关系 | 后续再定 |
| 非实时并发写冲突 | ❌ 没有考虑到(多端同时存盘覆盖) | 待定 | 协同前的过渡期风险 |

### A.9 当前的真空白汇总(❌)

- **格式治理基建**:版本声明、向前兼容(未知保留)、降级矩阵、一致性语料 —— 完全没有,是 mgmd 相对现状最大的增量。
- **待定的空白**:导入外部 md 归一化、导出器、diff 粒度、非实时并发写冲突。(表格/列表往返已由阶段 0 语料验证,移出空白。)

# Agent 图片提交即结束设计

> 后续扩展：同日的《Agent 视频提交即结束设计》已把相同终止边界扩展到视频；本文中“视频不在范围”的表述仅记录图片阶段的原始决策。

## 背景

当前图片生成接口已经是异步任务接口：`generate_media` 会先持久化图片任务，再由服务端后台执行；需要供应商继续查询时，服务端 worker 会接管轮询。项目资源任务完成后，服务端会自动选择本次结果的第一张图片，并通过生成完成事件刷新前端资源与任务缓存。

Agent 工作流仍要求在提交图片后调用 `poll_generation_task` 等待完成，再进行选片和文档回写。这让 Agent run 被图片耗时占用，也重复承担了服务端已经具备的后台任务职责。

## 决策

图片 Agent run 的终点改为“生成任务已被服务端接受”：

1. Agent 完成目标识别、模型目录读取、统一生成设置确认和生成请求构造。
2. `generate_media` 返回图片 `taskId`，或 `generate_media_batch` 返回批次 ID 与已接受的子任务 ID 后，Agent 立即发送 final answer 并结束当前 run。
3. 同一 run 不再调用 `get_generation_task`、`list_generation_tasks`、`poll_generation_task`、`retry_generation_task` 或 `select_generation_asset`，也不再等待结果选片或执行依赖最终图片 URL 的 Markdown 回写。
4. 服务端后台生成、供应商状态轮询、资产落库、项目资源首图自动选中和完成通知保持现状。
5. 查询、重试和选片工具继续保留，供适用的非图片流程或用户后续明确发起的独立状态查询/人工处理使用。

该规则只作用于图片；视频既有后台策略和后续显式跟进能力不在本次范围内。

## 契约层级

仅修改可编辑的内置 Skill 不足以稳定实现该行为，因为用户可能保留旧版自定义 Skill 或 `TOOLS.md`。因此需要同时更新三层指令：

- `image-generation` Skill：把标准流程改成“提交并结束”，移除同一 run 的等待、选片和回写阶段。
- 官方 `TOOLS.md`：把通用“submitted 后轮询到完成”改成媒体类型分流，明确图片提交后结束。
- Generation MCP instructions 与生成工具描述：作为运行时不可由项目 Skill 覆盖的硬边界，明确图片任务接受后不得在当前 run 继续查询或处理结果。

最终回复只报告实际提交结果，包括单任务 `taskId`、批次 ID、已接受/提交失败的子项和后台状态；不能把“已提交”描述成“已生成完成”，也不能承诺当前回复会包含图片地址。

## 数据流

```text
Agent 确认图片设置
  -> generate_media / generate_media_batch
  -> 服务端持久化 submitted task
  -> Agent 回复 taskId / batchId 并结束 run
  -> 服务端后台执行图片生成
  -> 必要时 worker 轮询供应商任务
  -> 资产落库 + 项目资源首图自动选中
  -> completion event 刷新前端并通知用户
```

## 错误边界

- 请求未被接受（校验、授权或提交调用直接报错）时，不得声称任务已提交；Agent 可报告提交错误。
- 批次部分子项提交失败时，已接受子项继续后台运行；final answer 同时列出接受数和失败子项，但不等待成功子项完成。
- 任务被接受后的供应商失败由生成任务状态记录。本次不新增失败 SSE、失败通知或 Agent 自动续跑。
- 启用提示词优化时，优化仍是提交调用的一部分；只有图片任务被实际接受后，Agent 才结束。

## 用户体验边界

项目资源携带 `documentContext` 时，后台完成会自动选择第一张图片，因此资源封面/定稿图不依赖 Agent 再调用选片工具。多图的人工改选由用户稍后在生成工作台完成。

自由生成或要求把最终 URL 插入 Markdown 的任务不会在同一 run 自动完成后处理；如果未来要支持这类连续动作，应新增服务端回写或任务完成后续跑机制，而不是让当前 Agent 长时间轮询。

## 非目标

- 不修改图片生成服务、后台 worker、任务状态模型或前端生成页面。
- 不新增失败通知、SSE replay 或生成中乐观状态。
- 不移除查询、轮询、重试和选片 MCP 工具。
- 不改变视频、音频或文本生成流程。

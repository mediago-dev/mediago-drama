# Agent 视频提交即结束设计

## 背景

视频生成接口已经采用任务化设计：异步视频路由会先创建本地任务，再由服务端后台提交供应商任务；后台 worker 继续同步状态、缓存视频资产并发送完成通知。携带项目资源 `documentContext` 的任务完成后，服务端还会自动选中本次生成的第一条视频。

当前 Agent 指令虽然建议视频任务进入后台后结束回合，但仍保留“同一回合内跟进”的轮询入口，以及任务完成后的重试、选片和文档回写步骤。因此 Agent 仍可能在一次视频生成 run 中等待分钟级任务完成，与接口的异步职责重复。

## 决策

视频 Agent run 的终点改为“生成任务已被服务端接受”：

1. Agent 完成目标识别、模型目录读取、参数确认、参考素材整理和生成请求构造。
2. `generate_media` 返回视频 `taskId`，或 `generate_media_batch` 返回批次 ID 与已接受的子任务 ID 后，Agent 立即发送 final answer 并结束当前 run。
3. 同一 run 不再调用 `get_generation_task`、`list_generation_tasks`、`poll_generation_task`、`retry_generation_task` 或 `select_generation_asset`，也不展示结果选片卡、不等待尾帧、不执行依赖最终视频的 Markdown 回写。
4. 服务端继续负责供应商提交、任务状态同步、资产落库、项目资源首条结果自动选中和完成通知。
5. 查询、重试和选片工具继续保留，供音频/文本等适用流程，或用户在后续 run 中明确查询、重试或改选既有任务时使用。

这与图片的“提交即结束”规则统一，但不改变音频和文本的现有流程。

## 服务端异步边界

当前生产目录中的 19 条视频路由全部标记为异步。`ShouldSubmitGenerationInBackground` 会先持久化本地 `submitting` 或 `queued` 任务，再启动后台供应商提交；HTTP/MCP 不等待供应商提交完成就返回本地 `taskId`。`generate_media_batch` 也只等待各子项被本地接受，不等待供应商任务或成片。

现有阻塞 provider 回归测试已经证明异步视频响应不会等待供应商提交完成，后台 worker 也会继续轮询、落库并发送视频完成通知。为让未来可能新增的同步视频路由同样满足契约，`ShouldRunGenerationInBackground` 也覆盖 `KindVideo && !Async`：服务端先创建本地任务并返回，再在后台执行完整供应商调用。启用 `promptOptimization` 时，文本提示词优化仍是视频提交前的同步准备步骤；视频任务一旦提交，仍立即返回并结束当前 run。

## 契约层级

需要同时更新四层边界：

- `video-generation` Skill：把“等待任务完成”和“交付并回写”改成“确认提交并结束”；试片也必须作为一次独立提交结束当前 run。
- 官方 `TOOLS.md`：统一图片和视频的提交终止语义，消除视频可以在同一 run 继续轮询的冲突。
- Generation MCP instructions 与工具描述：把图片/视频当前提交 run 的禁止查询、轮询、重试、选片和回写设为不可由旧 Skill 覆盖的硬边界。
- Generation service：让同步视频路由也在服务端后台执行，避免未来路由把生成耗时传回工具调用。

## 数据流

```text
Agent 确认视频设置
  -> generate_media / generate_media_batch
  -> 服务端持久化本地任务并返回 taskId / batchId
  -> Agent 报告初始状态并结束 run
  -> 服务端后台提交或执行供应商视频生成
  -> worker 同步供应商状态
  -> 视频资产落库 + 项目资源首条结果自动选中
  -> completion event 刷新前端并通知用户
```

## 试片与批量分镜

“先试片一组”不再允许同一 run 等待试片完成后继续批量：

1. 当前 run 只提交代表镜头试片并结束。
2. 试片完成后用户通过通知查看结果。
3. 用户在后续 run 明确确认后，再重新确认当轮有效的生成计划并提交剩余镜头批次。
4. 直接批量生成时，批次获得 ID 后同样立即结束，不汇总最终成功/失败数量。

这样不会复用已经跨 run 失效的 `confirmationSelectionId`，也不会让 Agent 为试片占用整段视频生成时间。

## 错误边界

- 请求未被接受（校验、授权、建任务或提交工具直接报错）时，不得声称任务已提交；Agent 只报告即时错误。
- 批次部分子项未被接受时，已接受子项继续后台运行；final answer 同时列出接受结果和即时失败子项，但不等待成功子项完成。
- 任务被接受后的供应商失败由任务状态和后续通知/查询处理；当前 run 不自动重试。
- 提示词优化仍属于提交前处理；只有视频任务被实际接受并取得任务 ID 后，当前 run 才结束。

## 最终回复

视频生成提交 run 只报告已经发生的事实：单任务 `taskId`，或批次 ID 与已接受的子任务 ID；初始后台状态；即时提交错误；以及完成后会通知。不得把“已提交”描述成“已完成”，不得给出尚不存在的视频地址、尾帧或文档回写位置。

## 非目标

- 不移除任务查询、轮询、重试和选片工具。
- 不修改生成任务状态模型、完成事件协议或前端生成历史页面。
- 不新增失败 SSE、任务完成后的 Agent 自动续跑或服务端 Markdown 回写。
- 不改变音频和文本生成流程。
